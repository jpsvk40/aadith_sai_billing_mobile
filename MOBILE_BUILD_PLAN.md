## PHASE 0 ‚Äî Backend: R2 object storage for scanned images (do first)
**Why:** `scannedInvoiceUrl` currently stores the whole image as **inline base64 in the DB row** (~1.6 MB each; ~418 MB total on prod). This caused the 64 MB Neon / OOM problem we already fixed in the reports. Mobile capture makes it worse. This is a **shared backend change that benefits web too** (web just gets a URL instead of a data URL).

Tasks:
1. **R2 bucket (private)** + upload helper in backend. On scan/upload: push bytes to R2, store only the key/URL in `scannedInvoiceUrl` (and the new credit-note `scannedCreditNoteUrl`).
2. **Serve securely (financial docs):** either
   - (a) **short-lived presigned GET URLs** generated by backend on demand *(recommended)*, or
   - (b) auth proxy `GET /api/.../scanned-image/:id` that checks JWT and streams from R2.
   Do **not** make the bucket public.
3. **Dual-mode during migration:** image-serving reads "if value starts with `data:` ‚Üí serve inline (legacy); else treat as R2 key." Nothing breaks mid-backfill.
4. **One-time backfill script:** read each base64 `scannedInvoiceUrl` ‚Üí upload to R2 ‚Üí replace column with key. Run in background.
5. **Web UI impact = tiny:** `<img src={...}>` / "View Scanned Bill" renders an https R2 URL same as a data URL ‚Äî plus removes base64 bloat from every vendor-purchase query.
6. Decide provider: **Cloudflare R2** (chosen direction) vs S3/Supabase. Configure CORS for browser/mobile if serving directly.

Also (infra, not strictly R2): add `connect_timeout=30` / keep-warm so the app's first call after Neon idle doesn't `P1001`.

## Lessons carried from the web work (apply on mobile)
- **Never send blob columns to a list endpoint** (caused OOM). Mobile lists must be lean + paginated.
- **Migrations apply on Render deploy**, but cold-start `P1001` can block the deploy ‚Äî warm Neon / retry / add connect_timeout.
- AI extracts **amounts** reliably but **identifiers (invoice no., GSTIN, year)** can be OCR-noisy ‚Üí keep the date-year + mismatch guards.

## E2E TESTING ‚Äî on the running emulator (Pixel 7 ¬∑ API 35)
**Harness:** Flutter `integration_test` (add to dev_dependencies) + `flutter_test`. Tests live in
`integration_test/` (e.g. `technician_flow_test.dart`, `admin_service_flow_test.dart`). Each drives the **real
app** widget tree end to end (login ‚Üí screens ‚Üí API calls ‚Üí assertions on visible widgets).

**Backend target (key detail):** the Android emulator reaches the host machine at **`10.0.2.2`**, so point the
app at the **local TEST backend**: run the backend (`node --experimental-require-module backend/src/server.js`,
TEST Neon in `backend/.env`), seed it (PHASE S0 #3), and launch the app with
`--dart-define=API_BASE=http://10.0.2.2:3001` (or a `.env.test`). E2E asserts against this seeded data ‚Äî the
same TEST DB the web service E2E uses. *(Optional: a Dio-interceptor "mock mode" returning fixtures for pure-UI
runs with no network ‚Äî add only if network flakiness bites.)*

**Pre-reqs / commands**
```
# emulator already running; verify (adb is in <AndroidSdk>/platform-tools)
adb devices                       # expect: emulator-5554  device
# backend up + seeded with a technician + an assigned ticket (PHASE S0 #3)

  --dart-define=API_BASE=http://10.0.2.2:3001
```

**Scenarios to cover**
- *Technician:* login (`service-e2e-tech@example.com`) ‚Üí My Tickets non-empty ‚Üí open ticket ‚Üí change
  **status** (asserts new status chip) ‚Üí add a **part** (asserts total updates) ‚Üí record **payment**

first (HTTP-against-live can be flaky), promote to required once stable. Mirror the web gate discipline:
